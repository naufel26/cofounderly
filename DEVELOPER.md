## 1. Overview

Cofounderly is a professional networking platform for Bangladesh's startup ecosystem — a LinkedIn-style SPA where founders, co-founders, investors, students, job seekers, and advisors can:

- Build profiles and showcase their startup journey
- Post updates and engage with a social feed
- Connect with others and manage a professional network
- Chat in real-time via private conversations
- Share ephemeral status stories (Instagram-style)
- View who has visited their profile
- Receive in-app notifications for social actions

The application is a **monolithic Laravel + Inertia SPA**: the server handles all routing and data, while React renders the UI client-side with no full-page reloads after the initial load.

## 3. Local Development Setup

### Prerequisites

- PHP 8.4+ with extensions: `pdo_mysql`, `mbstring`, `openssl`, `tokenizer`, `xml`, `ctype`, `json`, `bcmath`
- Composer 2+
- Node.js 20+ and npm
- MySQL / MariaDB (the app uses MariaDB in development)

### Steps

```bash
# 1. Clone the repository
git clone <repo-url> cofounderly
cd cofounderly

# 2. Install PHP dependencies
composer install

# 3. Copy environment file and generate app key
cp .env.example .env
php artisan key:generate

# 4. Configure your database in .env (see §4 below)
#    Set DB_CONNECTION, DB_HOST, DB_DATABASE, DB_USERNAME, DB_PASSWORD

# 5. Run migrations and seed roles
php artisan migrate
php artisan db:seed

# 6. Create the storage symlink
php artisan storage:link

# 7. Install JS dependencies and build assets
npm install
npm run build
# or for hot-reload development:
# npm run dev

# 8. Start all services (Laravel, queue, Reverb, Vite) in one command
composer run dev
```

`composer run dev` runs the following concurrently (see `composer.json scripts`):
- `php artisan serve` — HTTP server on `localhost:8000`
- `php artisan queue:listen --tries=1` — processes queued jobs
- `php artisan reverb:start` — WebSocket server on port `8080` (default)
- `npm run dev` — Vite HMR dev server

### Verify the Install

Open `http://localhost:8000` — you should see the Cofounderly landing page.

| POST | `/two-factor-challenge` | Submit TOTP code |

### Custom Registration Flow

Cofounderly uses a multi-step onboarding-aware sign-up instead of Fortify's default registration.

| Method | Path | Controller | Description |
|---|---|---|---|
| GET | `/signup` | (inline) | Renders `auth/signup.tsx` — multi-step form |
| POST | `/user-register` | `UserRegisterController@store` | Creates user, assigns Spatie role, redirects to `/onboarding` |

**`UserRegisterController@store` flow:**
1. Validates: `fullName`, `email`, `password`, `role`, `lookingFor` (array), `stage`, `interests` (array), `tagline` (optional)
2. Creates `User` record
3. Calls `Role::firstOrCreate(['name' => $request->role, 'guard_name' => 'web'])` to ensure the role exists
4. Assigns the role via `$user->assignRole($request->role)`
5. Logs the user in with `Auth::login($user)`
6. Redirects to `route('onboarding')`

> **Important:** The `RoleSeeder` must have been run before user creation (`php artisan db:seed`). The `firstOrCreate` call is a safety net but is not a substitute for seeding.

### Two-Factor Authentication

Managed by Fortify + `TwoFactorAuthenticatable` trait on `User`. Settings UI is at `resources/js/pages/settings/two-factor.tsx`.

## 7. Onboarding Flow

New users are routed through an onboarding page before accessing the main app.

### How It Works

1. `onboarding_completed_at` is `NULL` on a fresh user account.
2. The `EnsureOnboarded` middleware checks for `NULL` on every authenticated request and redirects to `GET /onboarding`.
3. The onboarding page (`resources/js/pages/onboarding.tsx`) welcomes the user, lets them upload a profile photo, and shows quick-start tips.
4. Clicking "Start Exploring" posts to `POST /onboarding/complete`, which sets `onboarding_completed_at = now()` and redirects to `/feeds`.

### Middleware Exemptions

The following routes are **exempt** from the onboarding check so the onboarding page itself can upload files:

- `onboarding` — the page itself
- `onboarding.complete` — the completion POST
- `profile.avatar.update` — avatar upload
- `profile.cover.update` — cover photo upload
- `logout` — always accessible
- `verification.*` — email verification

### Factories

`UserFactory::definition()` sets `onboarding_completed_at => now()` so test users bypass onboarding. Use the `pendingOnboarding()` state to test onboarding-specific behaviour:

```php
User::factory()->pendingOnboarding()->create(); // onboarding_completed_at = null
```

## 8. Middleware Stack

### Global (applied to all web routes)

Configured in `bootstrap/app.php`:

| Middleware | Purpose |
|---|---|
| `EncryptCookies` | Encrypts cookies (excludes `appearance` and `sidebar_state`) |
| `HandleAppearance` | Reads the `appearance` cookie and sets light/dark mode |
| `HandleInertiaRequests` | Shares global props with every Inertia response |

### Route-Level Middleware

| Middleware | Applied To | Purpose |
|---|---|---|
| `auth` | All protected routes | Redirect unauthenticated users to `/login` |
| `EnsureOnboarded` | Main app routes | Redirect users with `onboarding_completed_at = null` to `/onboarding` |
| `verified` | (available, not yet applied globally) | Require email verification |

### `HandleInertiaRequests` — Shared Props

Every Inertia page receives the following props:

```ts
{
  name: string,                     // APP_NAME
  auth: { user: User | null },      // Authenticated user with appended attributes
  sidebarOpen: boolean,             // Persisted sidebar preference from cookie
  profile_views_count: number,      // Lazy — count of viewers for current user
  notifications: {                  // Lazy
    unread_count: number,
    items: Notification[]           // 15 most recent, with actor info and action URL
  }
}
```

Lazy props are only evaluated when the page component accesses them (Inertia v2 deferred props).

## 10. Feature Modules

### Feed (`FeedController`, `resources/js/pages/feeds.tsx`)

- Renders a paginated timeline of posts from all users (public feed, not connection-filtered).
- Each post shows: author avatar + name + role, content, media carousel, like/comment counts, action buttons.
- The `CreatePost` component supports text, image uploads, and share/repost.
- The `StatusBar` component at the top renders active stories (statuses) as circular avatar bubbles.

### Posts (`PostController`, `resources/js/pages/posts/show.tsx`)

- `store` — validates content/media, stores `Post` record, attaches `PostMedia` entries, dispatches notifications.
- `toggleLike` — creates or deletes a `Like` record (idempotent).
- `storeComment` — supports `parent_id` for nested replies.
- `show` — single post page with full comment thread; used for SEO-accessible direct post links.

**SEO on post pages:**
- `<Head>` sets Open Graph `article` type, post URL, and image (first media item or author avatar).
- Twitter card type is `summary_large_image` when media is present, `summary` otherwise.

### Profile (`ProfileController`, `resources/js/pages/profile/show.tsx`)

- `show` — current user's own profile; `is_own_profile = true`.
- `showUser($user)` — another user's profile; records a view via `ProfileView::updateOrCreate()` (skips self-views).
- `updateProfile` — updates bio, skills, experience, links, etc.
- `updateAvatar` — validates image, deletes old file, stores in `avatars/`, updates `avatar` column.
- `updateCoverPhoto` — validates image (max 5 MB), deletes old file, stores in `covers/`, updates `cover_photo` column.
- `viewers` — returns paginated `ProfileView` records ordered by `viewed_at` desc, with viewer's `profile_photo_url` attached.
- `renderProfile` — shared helper that assembles the Inertia props for both `show` and `showUser`.

**Cover photo display:**
- If the user has a `cover_photo`, it renders as an `<img>` tag.
- Otherwise falls back to a teal-to-amber CSS gradient (`from-[#2DAB94] to-[#F1B981]`).
- The upload button only appears on hover for `is_own_profile = true`.

**SEO on profile pages:**
- `<Head>` sets `og:type = profile`, `og:title`, `og:description` (bio or tagline), `og:image` (profile photo URL).
- Twitter card is `summary` type.

### Profile Views (`ProfileView`, `resources/js/pages/profile/viewers.tsx`)

- Every visit to `/profile/{user}` calls `ProfileView::updateOrCreate(['viewer_id'=>$id, 'profile_user_id'=>$user->id], ['viewed_at'=>now()])`.
- The unique constraint on `(viewer_id, profile_user_id)` deduplicates rows; repeat visits update `viewed_at`.
- The viewer count is shared as a lazy prop in `HandleInertiaRequests` and shown in the left sidebar.
- `/profile/viewers` renders a paginated list with relative timestamps, viewer avatars, and inline connect actions.

### Connections (`ConnectionController`, `resources/js/pages/connections/index.tsx`)

- Connection status is tracked in the `connections` table as `pending`, `accepted`, `blocked`, or `ignored`.
- `sendRequest` creates a `pending` record.
- `acceptRequest` sets `status = accepted`.
- `ignoreRequest` sets `status = ignored` (soft-dismiss, not blocking).
- `removeConnection` hard-deletes the record.
- The unique constraint on `(sender_id, receiver_id)` prevents duplicate requests.
- The `scopeWithoutConnectionTo` scope on `User` excludes users already connected in any status.

### Chat (`ChatController`, `resources/js/components/chat/`)

- Real-time private messaging via Laravel Reverb WebSocket server.
- `startConversation` finds or creates a `Conversation` and adds both users to the `conversation_user` pivot.
- `sendMessage` stores a `Message` record and broadcasts `MessageSent` event on the `conversation.{id}` channel.
- The frontend uses a floating `ChatOverlay` component triggered by custom DOM events (`open-chat`, `open-chat-user`).
- The `MessageButton` in user profiles/feed dispatches `open-chat-user` to open a chat without prop drilling.
- Messages are fetched with cursor-based pagination and merged into the existing list (Inertia v2 merge).

### Statuses (`StatusController`)

- Ephemeral stories with a configurable `expires_at` timestamp.
- The `activeStatus` relationship on `User` filters by `expires_at > now()`.
- The `StatusBar` component shows active statuses in the feed; `StatusViewer` renders a full-screen story modal.

### Notifications (`NotificationController`)

- Uses Laravel's built-in database notification driver (`notifications` table).
- `markRead` / `markAllRead` update `read_at` timestamps.
- Notification data payload includes: `type`, `title`, `actor` (id, name, avatar), `action_url`.
- The `NotificationDropdown` component in the top nav shows real-time unread badge and dropdown list.

### Search (`UserSearchController`)

- Single invokable controller.
- Queries users by `name` or `email` using a `LIKE %term%` (or Eloquent `where` chain).
- Returns JSON for the live `SearchDropdown` component in the top nav.

### Inertia Page Conventions

- Pages live in `resources/js/pages/` and must be default exports.
- Props are typed via `usePage().props as { ... }` or destructured from the function parameter.
- Navigation uses Inertia's `<Link href="...">` — never `<a>` for internal links, never React Router's `<Link to="...">`.
- Form submissions use `router.post()`, `router.patch()`, `router.delete()`, or `useForm()` from `@inertiajs/react`.
- The `<Head>` component from `@inertiajs/react` sets the page title and meta tags.

```ts
import { usePage } from '@inertiajs/react';
```

## 13. Real-time System (Reverb + Echo)

### Server: Laravel Reverb

Reverb is Laravel's first-party WebSocket server. Start it with:

```bash
php artisan reverb:start
```

Configuration lives in `config/reverb.php`, driven by `REVERB_*` environment variables.

### Client: Laravel Echo

Echo is bootstrapped in `resources/js/app.tsx`:

```ts
import Echo from 'laravel-echo';
import Pusher from 'pusher-js';

window.Pusher = Pusher;
window.Echo = new Echo({
    broadcaster: 'reverb',
    key: import.meta.env.VITE_REVERB_APP_KEY,
    wsHost: import.meta.env.VITE_REVERB_HOST,
    wsPort: import.meta.env.VITE_REVERB_PORT,
    wssPort: import.meta.env.VITE_REVERB_PORT,
    forceTLS: import.meta.env.VITE_REVERB_SCHEME === 'https',
    enabledTransports: ['ws', 'wss'],
});
```

### Broadcasting Channels (`routes/channels.php`)

| Channel | Type | Authorization |
|---|---|---|
| `App.Models.User.{id}` | Private | User ID must match authenticated user |
| `conversation.{conversationId}` | Private | User must be a member of the conversation |

### Chat Message Flow

1. User sends a message — `POST /chat/conversations/{id}/messages`
2. `ChatController@sendMessage` stores the `Message` and broadcasts `MessageSent` on `conversation.{id}`
3. The `X-Socket-ID` header is included in the request so Reverb uses `toOthers()` — the sender does not receive their own broadcast
4. The recipient's Echo listener receives the event and appends the message to their local state

## 15. Error Handling

Custom error pages are configured in `bootstrap/app.php` and rendered as standalone Blade views (no Vite dependency, so they work even if assets fail to load).

### HTTP Status Code → View Mapping

| Status | View | Notes |
|---|---|---|
| 404 | `resources/views/errors/404.blade.php` | Brand-styled, teal gradient |
| 419 | `resources/views/errors/419.blade.php` | CSRF token expiry, amber gradient |
| 500 / 503 | `resources/views/errors/500.blade.php` | Server error, red gradient |

Custom error pages only activate when `APP_ENV` is not `local` or `testing`:

```php
if (!app()->environment(['local', 'testing']) && in_array($response->getStatusCode(), [500, 503])) {
    return response()->view('errors.500', [], $response->getStatusCode());
}
```

All error pages include a "Back to Home" link to `/`.

### Test Database

Tests use a separate in-memory or test database. Configure `phpunit.xml` for your setup. The `RefreshDatabase` trait (or Pest's `uses(RefreshDatabase::class)`) resets the database between tests.

### Factory States

| State | Effect |
|---|---|
| `User::factory()->create()` | Standard user, `onboarding_completed_at = now()` |
| `User::factory()->pendingOnboarding()->create()` | `onboarding_completed_at = null` |
| `User::factory()->unverified()->create()` | `email_verified_at = null` |
| `User::factory()->withTwoFactor()->create()` | 2FA configured and confirmed |

### Example Tests (`tests/Feature/OnboardingTest.php`)

```php
test('unauthenticated user is redirected from onboarding', function () {
    $this->get('/onboarding')->assertRedirect('/login');
});

test('already onboarded user is redirected from onboarding to feeds', function () {
    $user = User::factory()->create(); // onboarding_completed_at set by factory
    $this->actingAs($user)->get('/onboarding')->assertRedirect('/feeds');
});

test('new user can complete onboarding', function () {
    $user = User::factory()->pendingOnboarding()->create();
    $this->actingAs($user)->post('/onboarding/complete')->assertRedirect('/feeds');
    expect($user->fresh()->onboarding_completed_at)->not->toBeNull();
});
```

## 17. Code Style & Tooling

### PHP — Laravel Pint

Pint enforces PSR-12 style. Run before committing:

```bash
vendor/bin/pint --dirty --format agent
```

### TypeScript / React — ESLint + Prettier

```bash
npm run lint        # ESLint check
npm run format      # Prettier format
```

### Key Conventions

- **PHP**: Always use curly braces for control structures. Use constructor property promotion. Return type declarations on all methods. PHPDoc blocks with array shapes where appropriate.
- **Controllers**: Use Form Request classes for validation — never inline `$request->validate()` in complex controllers.
- **Eloquent**: Use relationships and eager loading. Avoid `DB::` raw queries unless absolutely necessary. Use `Model::query()` for explicit query building.
- **Routes**: Use named routes. Generate URLs with `route('name')` in PHP and Wayfinder functions in TypeScript.
- **Frontend**: Inertia `<Link href>` for navigation, never `<a>` for internal links. `router.post/patch/delete` for mutations. Never use React Router.
- **Comments**: No inline comments unless logic is non-obvious. No multi-line docstrings.

## 18. Deployment Checklist

```bash
# 1. Pull latest code
git pull origin main

# 2. Install/update PHP dependencies (no dev)
composer install --no-dev --optimize-autoloader

# 3. Install/build JS assets
npm ci
npm run build

# 4. Run database migrations
php artisan migrate --force

# 5. Seed roles (idempotent — uses firstOrCreate)
php artisan db:seed --class=RoleSeeder --force

# 6. Create/refresh storage symlink
php artisan storage:link

# 7. Cache config, routes, and views for production
php artisan optimize

# 8. Restart queue workers
php artisan queue:restart

# 9. Start / restart Reverb
php artisan reverb:start --daemon  # or use a process manager (Supervisor)

# 10. Restart PHP-FPM / web server
sudo systemctl reload php8.4-fpm
sudo systemctl reload nginx
```

### Environment Checklist for Production

- [ ] `APP_ENV=production`
- [ ] `APP_DEBUG=false`
- [ ] `APP_URL` set to the real domain (with HTTPS)
- [ ] `BROADCAST_CONNECTION=reverb`
- [ ] All `REVERB_*` and `VITE_REVERB_*` variables set
- [ ] `FILESYSTEM_DISK=public` and storage symlink created
- [ ] `MAIL_MAILER` set to a real driver (SMTP / SES / Mailgun)
- [ ] `QUEUE_CONNECTION=database` (or Redis for higher throughput)
- [ ] Database backed up before running `migrate --force`
- [ ] Reverb running under Supervisor or equivalent process manager

### Queue Workers

The application uses database-backed queues for notifications and other async jobs. In production, manage the worker with **Supervisor**:

```ini
[program:cofounderly-worker]

autostart=true
autorestart=true
```
